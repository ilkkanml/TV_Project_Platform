# M42 EA0 Critical Activation Key Generation / Hashing Plan

Status: Active Draft
Mode: Critical single-task planning only. No production deploy, no live secrets, no payment enforcement, no reseller workflow.

## 1. Purpose

M42 isolates the Activation Key generation and hashing plan.

This is a critical single task because Activation Key handling controls:

- Device access identity.
- License check trust.
- Future paid license continuity.
- Reinstall recovery safety.
- Secret leakage prevention.

## 2. Final Decision

EA0 uses this model:

```txt
Backend generates Device ID
Backend generates Activation Key
Backend stores activationKeyHash only
Backend may store activationKeyHint only
App stores raw Activation Key locally
License check verifies Activation Key against stored hash
```

The database must never store the raw Activation Key.

The APK must never contain a shared/global Activation Key.

## 3. Activation Key Generation Rules

Activation Key must be:

- Generated by backend.
- Unique enough for real users.
- Strong random value.
- Not predictable.
- Not based on Device ID only.
- Not based on timestamp only.
- Not generated by the app as final authority.

Recommended display format:

```txt
NX-ACT-XXXX-XXXX-XXXX-XXXX
```

Internal random source must be stronger than the human-readable grouping.

## 4. Device ID Generation Rules

Device ID should be:

- Backend-generated.
- Public-safe identifier.
- Unique.
- Stable for the device record.
- Usable in future customer portal.

Recommended display format:

```txt
NX-DEV-XXXX-XXXX
```

Rules:

- Device ID is not secret.
- Device ID alone must not grant access.
- Activation Key is the secret.

## 5. Hash Storage Rule

Database stores:

```txt
activationKeyHash
activationKeyHint
```

Database never stores:

```txt
activationKey
rawActivationKey
plainActivationKey
```

`activationKeyHint` may contain only a masked hint such as:

```txt
last 4 characters
```

## 6. Hashing Direction

Implementation must use a password/secret hashing approach suitable for stored access secrets.

Requirements:

- Hash raw Activation Key before database write.
- Verify submitted Activation Key using safe compare flow.
- Do not return hash to client.
- Do not log raw key or hash.
- Keep hashing configuration in environment/config, not hardcoded into public docs.

Acceptable implementation choices may include mature backend hashing libraries.

The final library choice belongs to backend implementation.

## 7. Create Flow

On first bootstrap:

```txt
1. Generate Device ID
2. Generate Activation Key
3. Create activationKeyHash
4. Create activationKeyHint
5. Store DeviceAccessRecord
6. Return Device ID + raw Activation Key once
```

Rules:

- Raw Activation Key is returned only in the bootstrap response.
- Raw Activation Key is not written to database.
- Raw Activation Key is not written to logs.

## 8. Recovery Rotation Flow

On reinstall recovery:

```txt
1. Match existing record by platformDeviceHash
2. Keep existing Device ID
3. Generate new Activation Key
4. Replace activationKeyHash
5. Replace activationKeyHint
6. Set lastRecoveredAt
7. Return new raw Activation Key once
8. Old Activation Key becomes invalid
```

Reason:

- Lost local credentials should not remain silently trusted forever.
- Recovery should continue the same Device ID when possible.

## 9. Verification Flow

On license check:

```txt
1. Receive Device ID + Activation Key
2. Find DeviceAccessRecord by Device ID
3. Verify Activation Key against activationKeyHash
4. Return generic denial if verification fails
5. Never reveal whether Device ID or Activation Key was the failed part
```

Generic error:

```txt
invalid_device_credentials
```

## 10. Logging and Audit Redaction

Never log:

```txt
activationKey
activationKeyHash
full request body containing activationKey
```

Audit may record:

```txt
device_access.created
device_access.recovered
device_access.key_rotated
license.checked
license.denied
```

Audit metadata may include:

```txt
deviceId
platform
appVersion
result category
```

Audit metadata must not include raw secrets.

## 11. Rate Limit Requirement

Rate limits must protect:

```txt
POST /devices/bootstrap
POST /license/check
```

Reason:

- Prevent guessing attempts.
- Reduce abuse during public download-only early access.

Client response must not expose threshold details.

## 12. Future Paid Compatibility

Activation Key must remain valid across free-to-paid transition unless reset is needed.

Do not require by default:

```txt
new Device ID
new Activation Key
app reinstall
customer email/name account
```

Paid license state changes should happen on backend records, not by replacing EA0 identity.

## 13. Stop Conditions

Stop immediately if implementation tries to add:

- Raw Activation Key database field.
- Raw Activation Key logs.
- Shared/global key inside APK.
- App-generated permanent key as final authority.
- Device ID-only access.
- Customer email/name requirement for EA0.
- MAC as primary product identity.
- Payment blocking during EA0.

## 14. Acceptance Criteria

M42 is acceptable when:

- Activation Key generation is backend-owned.
- Activation Key is strong and non-predictable.
- Database stores hash only.
- Activation Key hint is masked only.
- Create/recovery/license-check flows are clear.
- Logging and audit redaction are clear.
- Future paid licensing can reuse the same record.
