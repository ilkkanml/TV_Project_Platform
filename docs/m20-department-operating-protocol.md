# M20 Department Operating Protocol

Status: Active
Mode: ChatGPT-led planning, documentation, memory, and department coordination.

## 1. Purpose

M20 defines how departments operate inside the project without creating unnecessary document bloat, repeated context, or scope drift.

The goal is fast movement with controlled memory.

## 2. Current Operating Mode

The project currently runs through ChatGPT as the main coordination workspace.

Active work:

- Planning.
- Documentation.
- Department routing.
- Scope control.
- Project memory.
- Checkpoints.
- Specialist task packs.

Paused work:

- Hosting.
- Railway.
- Live database setup.
- Production deploy.
- Heavy backend implementation.
- Heavy Android implementation.

## 3. Department Philosophy

Departments are not separate products.

They are specialist lenses used only when useful.

A department should be called when:

- The decision needs expert framing.
- A risk area needs focused review.
- A milestone needs specialist instructions.
- A future implementation pack must be prepared.

A department should not be called for:

- Small wording edits.
- Simple checkpoint updates.
- Repeating already-approved direction.
- Creating fantasy structure with no decision value.

## 4. Central Coordination Role

The central coordinator owns:

- Master project memory.
- Document map.
- Department routing.
- Scope drift detection.
- Milestone checkpointing.
- Cross-project alignment.
- Final decision summary.

The central coordinator does not override approved contracts without creating a new explicit decision record.

## 5. Required Context Before Department Work

Every department prompt must include:

- Project name.
- Current mode.
- Relevant milestone.
- Relevant documents.
- No-content boundary.
- Free launch rule.
- Hosting/deploy pause rule.
- Required output format.
- Stop conditions.

This avoids repeating the entire project history.

## 6. Department Output Rules

Department output should be:

- Specific.
- Decision-oriented.
- Scope-aware.
- Additive.
- Easy to convert into a project record.

Department output should avoid:

- New feature spam.
- Host/deploy assumptions.
- Content-provider behavior.
- Payment-first pressure.
- Reseller-first expansion.
- Huge rewrites.
- Unrequested implementation code.

## 7. Department Categories

Active specialist categories:

- Ecosystem Integration Director.
- Backend Specialist.
- Android Specialist.
- Security Specialist.
- Auth Specialist.
- Dashboard / UX Specialist.
- Documentation / Memory Coordinator.

Optional later categories:

- Payment Specialist.
- Reseller Specialist.
- Legal/Compliance Reviewer.
- QA/Test Specialist.
- Release Manager.

Deferred until needed:

- Marketing.
- Analytics.
- Growth.
- Advanced monetization.
- Live operations.

## 8. Escalation Rules

Escalate to Ecosystem Integration Director if:

- Platform and Android needs conflict.
- A new endpoint is proposed.
- A database field risks content-provider drift.
- Payment enforcement appears before free launch ends.
- Dashboard expands into source/channel management.
- A specialist requests host/deploy work during pause.

Escalate to Security Specialist if:

- Secrets, tokens, logs, auth, profile transfer, or rate limits are involved.

Escalate to Auth Specialist if:

- Login, register, roles, device ownership, activation approval, or admin access are involved.

Escalate to Dashboard Specialist if:

- Customer/admin UI scope changes.

## 9. Stop Conditions

Any department must stop if a task asks for:

- Content hosting.
- Channel selling.
- Stream relay.
- Stream transcoding.
- Provider scraping.
- Provider credential storage.
- DRM bypass.
- Unauthorized source collection.
- Payment-first blocking during free launch.
- Hosting/deploy/live database work without explicit re-approval.

## 10. Document Creation Rule

Create a new document only when:

- It records a milestone.
- It defines a reusable rule.
- It gives a specialist task pack.
- It prevents future confusion.
- It becomes a handoff source.

Do not create a document for:

- Tiny notes.
- Temporary thoughts.
- Duplicate summaries.
- Reworded versions of existing decisions.

## 11. Checkpoint Rule

After each major decision, record:

- What changed.
- What is now approved.
- What is paused.
- What is deferred.
- Which document controls the decision.
- Next recommended step.

Checkpoints should be short and direct.

## 12. Prompt Routing Rule

When the user says `devam`, continue the next safe planning/checkpoint step.

When the user says `departmana gönder`, prepare a specialist prompt.

When the user says `kayıt altına al`, create or update the relevant project document.

When the user says `dur`, pause the current chain and summarize current status.

When the user says `kod yok`, produce planning only.

When the user says `host yok`, avoid deployment, Railway, database migration, or production setup tasks.

## 13. Specialist Prompt Format

Use this structure for specialist prompts:

1. Project context.
2. Current status.
3. Relevant documents.
4. Task.
5. Focus areas.
6. Rules / forbidden scope.
7. Required output format.
8. Stop conditions.

The prompt should be short enough to use, but complete enough to prevent drift.

## 14. Memory Hygiene

Project memory should prioritize:

- Active decisions.
- Current mode.
- Approved boundaries.
- Active milestones.
- Next steps.
- Stop conditions.

Project memory should not preserve every small conversation detail if it does not affect future decisions.

## 15. Lightweight Work Rule

Prefer small, clear records over giant documents.

Each document should answer one job:

- What is the decision?
- Who uses it?
- What is allowed?
- What is forbidden?
- What happens next?

## 16. Department Independence Rule

Departments may recommend direction, but they do not silently change project law.

A department recommendation becomes active only after:

- It is accepted by the central coordinator/user.
- It is recorded in the correct checkpoint or contract.
- It does not conflict with higher-priority documents.

## 17. Priority Order

When documents conflict, priority is:

1. Master Project Checkpoint.
2. M10 Ecosystem Alignment Contract.
3. Ecosystem Integration Role.
4. Current milestone contract.
5. Specialist task pack.
6. Department suggestion.
7. Temporary chat note.
